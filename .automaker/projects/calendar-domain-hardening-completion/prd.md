# PRD: Calendar Domain Hardening & Completion

## Situation
The calendar domain is structurally complete (types, service, routes, MCP tools, UI, job executor, Google Calendar sync) but a thorough code audit identified 14 issues spanning P1 bugs, security vulnerabilities, broken pipelines, missing integrations, and zero test coverage. The system is operational but unreliable in multi-project hivemind mode, has a dead reminder-to-reactive-spawn pipeline, silently ignores cancelled Google events, exposes a shell injection surface, and the calendar-assistant skill references execute_dynamic_agent which is being deprecated.

## Problem
1. MCP tool type enum uses 'feature_due_date' instead of 'feature', missing 'google' and 'ceremony' — type-based filtering is broken for MCP callers. 2. emitReminder() is wired in services.ts but never called by JobExecutorService — the reminder→ReactiveSpawner pipeline is dead code. 3. JobExecutorService.executeCommand() passes user-provided strings directly to exec() — shell injection vulnerability. 4. CRDT uses a single global 'shared' document for all projects — multi-project calendar data collides in hivemind mode. 5. Calendar mutations emit no WebSocket events — UI polls, other instances don't get real-time updates. 6. CalendarEventType includes 'ceremony' but ceremony-service.ts has no CalendarService integration — ceremony events never appear on the calendar. 7. calendar-assistant skill references execute_dynamic_agent which is being removed. 8. Google sync skips cancelled events but never deletes previously synced events that were later cancelled upstream. 9. ID prefix inconsistency between createEvent() and upsertBySourceId() paths. 10. No periodic Google Calendar sync — manual-only. 11-14. No tests for any calendar code.

## Approach
Three milestones in dependency order: (1) Fix all P1 bugs and security issues first — enum mismatch, shell injection, CRDT scoping, dead reminder wiring. (2) Add missing runtime integrations — WebSocket push for calendar events, ceremony→calendar bridge, Google sync cleanup and scheduling. (3) Rewrite calendar-assistant skill for SDK-native patterns and add test coverage for the now-correct implementation.

## Results
All 14 audit issues resolved. MCP type filtering works correctly. emitReminder() is called on job completion/failure and the reactive spawn pipeline activates. Shell injection is eliminated via command validation. CRDT calendar documents are scoped per-project. Calendar mutations broadcast WebSocket events to all connected clients. Ceremony events appear on the calendar. Cancelled Google events are cleaned up on sync. calendar-assistant skill is SDK-native. Tests cover CalendarService, JobExecutorService, and calendar routes.

## Constraints
Must not break existing calendar UI or existing job scheduling behavior,CRDT schema change (project-scoped docs) must not corrupt existing calendar.json filesystem fallback data,Shell injection fix must not break legitimate run-command job actions,ceremony-service.ts integration must not create circular service dependencies,Each phase must leave build + tests passing independently,No backward-compatibility shims — update all consumers immediately
